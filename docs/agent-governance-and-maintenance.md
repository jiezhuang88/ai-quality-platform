# 8 个 SDLC Agent 协同与维护方案

## 维护目标

8 个 Agent 不是一次性脚本，而是研发平台中的长期能力资产。维护目标是做到：职责清晰、输入输出可验证、权限可控、结果可追溯、版本可灰度、效果可评估。

## Agent 划分

| 阶段 | Agent | 核心职责 | 主要 Owner |
| --- | --- | --- | --- |
| 需求澄清 | 需求澄清 Agent | 将需求转成规则卡、验收标准、歧义清单 | 产品负责人 |
| 方案评审 | 方案评审 Agent | 识别系统影响面、服务依赖、接口契约和风险等级 | 架构负责人 |
| AI 编码 | Coding Agent | 生成代码、单测、自检说明和 PR 草稿 | 研发 Owner |
| PR 门禁 | PR Diff Agent | 识别 diff 风险、测试影响、安全与质量阻断项 | 研发 Owner + QA Owner |
| 测试验证 | 测试编排 Agent | 编排数据、自动化、性能、混沌和回归范围 | QA Owner |
| UAT 验收 | UAT Agent | 生成业务样例、组织验收、沉淀签收证据 | 业务验收 Owner |
| 发布决策 | 发布决策 Agent | 汇总证据，给出放行、灰度、阻断或例外评审结论 | 发布经理 |
| 复盘沉淀 | 复盘沉淀 Agent | 将线上结果和门禁表现沉淀为规则、知识和评测集 | 质量运营 |

## 协同机制

1. 上游 Agent 必须输出结构化资产，不能只输出自然语言结论。
2. 下游 Agent 只消费已签名的上游资产，例如需求规则卡、影响面分析、PR 风险摘要、测试报告。
3. 每次执行生成 `run_id`，所有 Agent 的输入、输出、工具证据和人工审批都挂到同一条执行链路。
4. 高风险节点设置人工确认点：业务规则、架构影响、核心代码、测试覆盖、例外发布、门禁规则变更。
5. 发布后由复盘沉淀 Agent 反哺需求规则库、风险模型、用例推荐和 Agent 评测集。

## 后台需要支持什么

- Agent Registry：维护 Agent ID、版本、职责、Owner、工具权限、输入输出契约、门禁规则、SLA、评估指标。
- Workflow Orchestrator：根据项目风险等级和阶段状态调度 Agent，控制串行、并行、阻断和重试。
- Evidence Store：存储需求、代码、测试、UAT、发布、监控和审批证据。
- Policy Engine：维护 L2/L3/L4 风险门禁规则，决定哪些证据必需、哪些问题可例外。
- Tool Adapter：对接数据银行、自动化框架、性能平台、混沌工程、Sonar、覆盖率、监控、发布系统。
- Human Approval：对高风险、例外发布、权限变更、规则变更保留人工审批。
- Observability：记录 Agent 成功率、误判率、耗时、重试、工具失败、门禁命中率。
- Evaluation：用历史缺陷、线上事故、典型 PR 和业务 Case 持续评测 Agent 质量。

## 维护节奏

- 每周：检查 Agent 失败率、误阻断率、漏检缺陷、工具调用失败。
- 每双周：更新提示词、规则库、用例推荐策略和风险阈值。
- 每月：基于线上事故和逃逸缺陷更新评测集。
- 每季度：复盘 Agent 权限、Owner、SLA、组织协同流程和平台 ROI。

## 当前系统实现

本地系统已在 `web-quality-workbench/server.py` 中维护 `Agent Registry`，并提供：

- `GET /api/agents`：查询 8 个 Agent 维护信息。
- `GET /api/agents/{id}`：查询单个 Agent。
- `PUT /api/agents/{id}`：保存 Agent 配置。
- `POST /api/agents/{id}/simulate`：基于当前项目群试运行 Agent。
- 项目阶段自动绑定对应 Agent。
- 前端展示 Agent 职责、输入输出契约、工具权限、门禁、评估指标和下游交接。
- 质量报告自动追加 `Agent Governance` 段落。

## 完整 Agent 维护字段

完整 Agent 至少包含以下字段：

- 基础定义：`id`、`name`、`stage`、`owner_role`、`version`、`status`。
- 任务边界：`mission`、`system_prompt`、`guardrails`、`human_checkpoint`。
- 契约：`input_schema`、`output_schema`。
- 权限：`tool_permissions`，包含工具名、访问级别、审批要求和作用范围。
- 策略：`policy_rules`，包含规则编号、规则内容和严重等级。
- 评测：`eval_cases`，用典型业务 Case 校验 Agent 是否会误放行或误阻断。
- 运维：`runtime`、`maintenance`、`versions`。

## 样板 Agent

当前系统已把 `测试编排 Agent` 建成完整样板，覆盖：

- 沃尔玛订单交易核心链路。
- 全场券和商品券组合测试，不引入平台券、店铺券等不存在概念。
- PR 风险摘要、影响面分析、质量指标、工具能力作为输入。
- 测试策略、数据计划、自动化计划、性能计划、混沌计划、证据缺口和门禁结论作为输出。
- 数据银行、自动化框架、性能平台、混沌工程、缺陷平台、监控平台作为工具权限。
- 支付前金额一致性、接口自动化失败归因、L3/L4 核心链路覆盖作为阻断策略。

## 维护方式

1. 打开本地系统 `http://127.0.0.1:8787/`。
2. 进入页面中的 `完整 Agent 维护台`。
3. 选择 `测试验证 · 测试编排 Agent`。
4. 修改 Owner、版本、系统提示词、Schema、工具权限、策略规则或评测用例。
5. 点击 `试运行`，基于当前项目群检查 Agent 输出是否符合预期。
6. 填写变更说明，点击 `保存配置`。
7. 后台会写入 `web-quality-workbench/data/system_state.json`，并记录版本变更。

## 支撑年度几百项目的方式

平台按三个层次承载大型研发组织：

1. 项目组合层：维护年度项目台账、季度节奏、风险等级、项目状态、角色清单和预计 Agent 执行次数。
2. 流程模板层：把大型项目统一映射到 8 个 SDLC 阶段，每个阶段绑定 Agent、责任角色和门禁标准。
3. 角色化 Agent 层：同一个 Agent 在不同角色视角下有不同审批点、责任边界和工具权限。

当前本地系统提供：

- `GET /api/portfolio`：查询年度项目组合、流程模板和角色化 Agent 配置。
- `POST /api/projects/intake`：接入新项目到年度质量保障台账。

### 年度容量模型

默认模板 `enterprise-sdlc-quality-template` 支持年度 360 个项目：

- L1：轻量门禁，适合低风险配置、页面、小改动。
- L2：标准门禁，适合普通业务迭代。
- L3：强门禁，适合交易、支付、库存、履约、算法灰度等核心链路。
- L4：最高门禁，适合资金、安全、数据回补、大促、核心架构改造。

每个项目都会绑定：

- 项目风险等级。
- 全流程 8 阶段模板。
- 角色清单。
- Agent 执行次数。
- 质量目标。
- 门禁策略。

### 角色化配置

平台内置 8 类角色：

- 产品 Owner：需求准确性、业务验收、歧义确认。
- 研发 Owner：方案可落地、代码质量、PR 风险控制。
- QA Owner：测试策略、证据完整性、失败归因。
- 架构 Owner：跨域依赖、架构风险、回滚可行性。
- 数据/算法 Owner：数据血缘、特征质量、模型灰度证据。
- 业务验收 Owner：UAT 样例、业务签收、遗留风险接受。
- 发布经理：发布窗口、灰度策略、回滚和审批。
- 质量运营：复盘、规则治理、Agent 评测集维护。

这些角色不是展示标签，而是决定：

- 哪些 Agent 能被调用。
- 哪些阶段必须审批。
- 哪些工具需要授权。
- 哪些证据缺失会阻断。
- 哪些项目必须升级到质量委员会。
