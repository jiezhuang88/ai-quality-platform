# 面向完整 SDLC 的 AI Coding 代码质量保障系统

## 系统定位

这套系统用于支撑 AI coding 进入规模化软件交付后的质量治理。它不是单一测试工具，而是一套覆盖完整软件开发生命周期的工程质量操作系统。

覆盖范围：

- 需求准入。
- 设计评审。
- AI 编码约束。
- PR 准入。
- 测试验证。
- 安全与供应链。
- 发布控制。
- 运行监控与缺陷复盘。

## 核心原则

1. AI 代码默认不可信，必须用证据证明可合并。
2. 风险等级决定门禁强度。
3. 自动化负责快速阻断，AI 负责辅助分析，人负责业务语义和风险裁决。
4. 每个阶段都必须有可审计证据。
5. 逃逸缺陷必须反哺测试、Prompt 和门禁规则。

## 生命周期控制面

| 阶段 | 主要问题 | 必须产出 |
| --- | --- | --- |
| 需求准入 | AI 是否理解正确业务目标 | 业务目标、验收标准、影响范围、风险评估 |
| 设计评审 | 是否存在架构/数据/安全风险 | 接口契约、回滚策略、安全设计评审 |
| AI 编码约束 | AI 改了什么、不能改什么 | AI 使用说明、Prompt 假设、禁止范围、人工复核 |
| PR 准入 | 是否可以进入合并流程 | PR 描述、改动文件、测试说明、风险等级 |
| 测试验证 | 是否按风险执行足够测试 | 单元、集成、契约、E2E、安全、AI 测试分析 |
| 安全供应链 | 是否引入安全或依赖风险 | SAST、密钥扫描、SCA、SBOM、LLM 风险评审 |
| 发布控制 | 是否具备上线保护 | 发布说明、回滚、监控、灰度、审批 |
| 运行复盘 | 是否持续改进质量系统 | 指标、回归反馈、线上监控、复盘报告 |

## 可执行组件

### 1. 策略文件

路径：

```text
policies/sdlc-quality-policy.json
```

用途：

- 定义每个 SDLC 阶段。
- 定义每个风险等级所需证据。
- 支撑自动化门禁判断。

### 2. 证据清单

样例：

```text
examples/sdlc/evidence-manifest.sample.json
```

团队每个 AI 代码变更都应生成一份 evidence manifest，记录该变更在各阶段的质量证据。

### 3. SDLC 门禁 CLI

命令：

```bash
python3 scripts/sdlc_gate.py --manifest examples/sdlc/evidence-manifest.sample.json
```

生成报告：

```bash
python3 scripts/sdlc_gate.py \
  --manifest examples/sdlc/evidence-manifest.sample.json \
  --report reports/sdlc-quality-report.md
```

### 4. Web 工作台

路径：

```text
web-quality-workbench/
```

用途：

- 录入变更。
- 风险评分。
- 查看控制框架。
- 记录测试、安全、供应链、AI 评审和度量证据。
- 导出质量报告。

## 自动化与 AI 的结合方式

### 自动化负责

- 单元测试。
- 集成/契约测试。
- E2E 测试。
- 静态扫描。
- 密钥扫描。
- 依赖扫描。
- 覆盖率检查。
- SBOM 生成。
- 发布监控检查。

### AI 负责辅助

- 分析 PR diff。
- 识别影响范围。
- 生成测试点。
- 找出边界条件。
- 建议回归范围。
- 分析失败日志。
- 总结风险和阻断建议。

### 人负责

- 判断业务语义是否正确。
- 判断风险等级是否合理。
- 审核 AI 生成测试质量。
- 对高风险例外发布做决策。

## 推进顺序

1. 所有 AI 代码必须进入 PR。
2. 所有 PR 必须声明 AI 参与范围。
3. 建立风险分级。
4. 接入基础 CI 门禁。
5. 接入 SDLC 证据清单。
6. 接入 AI diff 分析。
7. 对高风险链路建立强制 E2E 和安全测试。
8. 建立线上指标和逃逸缺陷复盘。

## 成功指标

- AI 代码 PR 证据完整率。
- 自动化门禁拦截率。
- 缺陷逃逸率。
- AI 变更失败率。
- 高风险变更 E2E 覆盖率。
- 安全扫描高危清零率。
- 平均修复时间。
- 回归通过率。
