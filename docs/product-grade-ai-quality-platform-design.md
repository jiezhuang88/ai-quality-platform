# AI Coding 时代研发质量保障平台产品级设计文档

版本：v1.0  
日期：2026-06-11  
定位：面向大型互联网研发组织的 SDLC 全流程质量保障平台，支撑年度数百项目、多角色协同、AI Coding 大量代码提交下的质量门禁与证据闭环。

## 1. 建设目标

当前系统不能停留在 Demo 层，需要建设为可长期运营的产品级平台。平台核心目标是：

1. 支撑大型项目从需求、设计、AI 编码、PR、测试、UAT、发布到复盘的全流程质量保障。
2. 支持产品、研发、测试、架构、数据、算法、业务、发布等多角色协同。
3. 支持一年几百个项目的接入、分流、跟踪、门禁和度量。
4. 接入真实工具链：PRD 文档、代码仓库、PR、Sonar、单测覆盖率、自动化框架、测试数据平台、性能平台、混沌工程、缺陷平台、发布系统、监控系统。
5. 用 Agent 做“辅助决策与自动编排”，但关键门禁、例外发布、核心链路变更保留人工责任。

## 2. 产品边界

平台不是替代所有研发工具，而是做研发质量控制面。

### 平台负责

- 项目接入与风险分级。
- PRD/需求资产解析与结构化。
- SDLC 阶段流程编排。
- Agent 配置、调度、试运行、评测和版本维护。
- Sonar、覆盖率、自动化、性能、混沌、缺陷、监控等证据汇聚。
- 测试用例生成、评审、维护、执行映射。
- 质量门禁策略配置。
- 发布准入报告。
- 项目组合级度量与管理驾驶舱。

### 平台不直接负责

- 不替代代码仓库。
- 不替代 CI/CD 执行引擎。
- 不替代自动化测试框架本身。
- 不替代业务最终验收责任。
- 不绕过人工审批直接发布高风险项目。

## 5. 关键业务流程

### 5.1 新项目接入

入口：项目中心 -> 新建项目

### 5.1.1 大型企业项目建模原则

平台必须按“大型项目空间”建模，而不是按单个仓库或单个 PR 建模。一个项目通常会关联多个系统、多个代码仓库、多个团队、多个测试环境、多个发布单和多类工具证据。

项目空间需要承载：

- 项目基本信息：项目名称、业务域、项目类型、计划周期、风险等级、项目负责人。
- 业务范围：影响的业务流程、核心链路、用户群、数据范围、运营活动范围。
- 系统清单：涉及系统、系统 Owner、业务域、系统关键性、上下游关系。
- 仓库清单：多个 Git 地址、默认分支、代码 Owner、对应系统、CI Pipeline。
- 接口清单：新增/变更接口、契约文档、调用方、被调用方。
- 数据清单：涉及表、消息 Topic、数据回补任务、特征或指标。
- 环境清单：开发、测试、预发、灰度、生产环境以及可用性状态。
- 角色清单：产品、研发、QA、架构、数据/算法、UAT、发布、质量运营。
- 工具配置：Sonar 项目 Key、自动化 Suite、数据银行空间、性能项目、混沌实验、监控看板。
- 质量计划：风险等级、阶段门禁、必需证据、关键里程碑、阻断项。

项目与系统、仓库的关系：

```text
Project 1 - N System
System 1 - N Repository
Repository 1 - N Pull Request
Project 1 - N Requirement
Project 1 - N Test Plan
Project 1 - N Release Gate
```

这意味着：

- 新建项目时必须支持维护多个系统。
- 每个系统可绑定一个或多个 Git 仓库。
- PR 阶段必须能识别 PR 属于哪个系统和仓库。
- 风险评估不能只看单 PR，需要结合项目级影响面。
- 发布准入需要汇总整个项目空间下所有系统、仓库、PR、测试和审批证据。

必填信息：

- 项目名称
- 业务域
- 项目类型：业务迭代、架构改造、数据改造、算法灰度、稳定性专项
- 计划周期
- 项目负责人
- 产品 Owner
- 研发 Owner
- QA Owner
- 业务验收 Owner
- 涉及系统清单
- 关联代码仓库清单
- 核心接口或数据范围
- 工具链映射
- 风险初评

系统动作：

1. 创建项目空间。
2. 绑定 SDLC 流程模板。
3. 根据项目类型和风险等级推荐 Agent 组合。
4. 创建质量计划草稿。
5. 生成项目质量台账。

6. 为每个系统生成质量责任卡。
7. 为每个仓库建立 PR/Sonar/CI 证据映射。
8. 为 L3/L4 项目自动要求架构 Owner、发布经理和质量运营参与。

风险分级：

- L1：低风险，小改动。
- L2：普通业务迭代。
- L3：核心业务链路、交易、库存、履约、支付、算法灰度。
- L4：资金、安全、核心架构、数据回补、大促、重大稳定性变更。

## 6. PRD 导入与需求资产

### 6.1 支持的 PRD 来源

第一阶段支持：

- 本地 Markdown
- 本地 Word/PDF
- 粘贴文本
- URL 链接登记
- 飞书/Confluence/Jira 链接登记
- 多文档组合导入：主 PRD、接口文档、埋点文档、UAT 样例、技术方案可关联同一需求空间

第二阶段接入：

- 飞书文档 API
- Confluence API
- Jira Issue API
- Aha/禅道/自研需求平台

### 6.1.1 PRD 导入产品能力要求

PRD 导入不能只是一个文本框，需要具备产品级能力：

- 支持多来源：文件上传、粘贴文本、普通 URL、飞书文档、Confluence、Jira、需求平台。
- 支持多文件：一个项目可导入多个 PRD 或补充文档。
- 支持版本管理：同一 PRD 更新后生成新版本，保留差异。
- 支持解析状态：待解析、解析中、解析成功、解析失败、待人工确认。
- 支持原文引用：Agent 抽取出的规则必须能追溯到原文段落或文档位置。
- 支持权限校验：飞书或 Confluence 链接需要校验访问权限。
- 支持人工修订：产品 Owner 可修改业务规则、验收标准和歧义点。
- 支持确认流：需求规则卡必须经产品 Owner 确认后进入后续阶段。
- 支持变更影响：PRD 更新后，自动提示受影响的测试用例、PR、自动化和 UAT 样例。

### 6.2 PRD 导入流程

1. 上传或登记 PRD。
2. 系统解析文档。
3. 需求澄清 Agent 抽取结构化资产：
   - 需求目标
   - 业务规则
   - 用户路径
   - 角色权限
   - 数据变更
   - 接口变更
   - 验收标准
   - 歧义问题
   - 风险初判
4. 产品 Owner 确认或修正。
5. 生成需求规则卡。

6. 建立需求到系统、接口、数据、测试用例的追踪关系。
7. 若 PRD 变更，触发需求变更影响分析。

### 6.3 需求资产数据结构

```json
{
  "requirement_id": "REQ-2026-0001",
  "project_id": "P-2026-001",
  "source_type": "docx",
  "source_url": "",
  "business_goal": "",
  "rules": [],
  "acceptance_criteria": [],
  "ambiguities": [],
  "risk_hints": [],
  "confirmed_by": "",
  "status": "draft|confirmed|changed"
}
```

## 7. PR 阶段质量门禁

### 7.1 PR 接入方式

支持：

- GitHub/GitLab/Bitbucket PR Webhook
- 手动登记 PR URL
- CI Pipeline 回调

### 7.1.1 多仓库 PR 管理

大型项目通常有多个仓库，PR 门禁必须支持项目级多仓库视图：

- 一个项目可关联多个仓库。
- 一个仓库可有多个 PR。
- PR 必须标识所属系统、仓库、分支、代码 Owner。
- PR 风险评估需要结合项目上下文，而不是只看当前仓库。
- 跨仓库变更需要形成联动门禁，例如订单服务和支付服务同时改动时，必须增加端到端测试。
- 发布准入需要汇总所有 PR 的门禁状态。

PR 维度需要记录：

- PR URL、仓库、分支、提交人、Review 人。
- 关联需求、关联系统、关联接口、关联测试计划。
- AI 生成代码比例和 AI 工具说明。
- Sonar、单测覆盖率、SAST/SCA、CI、自动化结果。
- 阻断项、例外审批、合入状态。

### 7.2 Sonar 集成

Sonar 不建议由平台直接执行扫描，建议由 CI 执行，平台汇聚结果。

推荐流程：

1. PR 创建。
2. CI 触发构建。
3. CI 执行：
   - 单测
   - 覆盖率
   - Sonar 扫描
   - SAST/SCA
4. CI 将结果回调平台。
5. 平台通过 Sonar API 拉取明细。
6. PR Diff Agent 汇总风险。

### 7.3 Sonar 对接字段

```json
{
  "projectKey": "",
  "pullRequest": "",
  "qualityGateStatus": "OK|ERROR",
  "bugs": 0,
  "vulnerabilities": 0,
  "codeSmells": 0,
  "coverage": 82.4,
  "duplicatedLinesDensity": 1.2,
  "blockerIssues": [],
  "criticalIssues": []
}
```

### 7.4 PR 门禁规则

L2：

- Sonar Quality Gate 必须通过。
- 单测必须通过。
- 覆盖率不得低于项目基线。

L3：

- L2 全部满足。
- 新增/修改核心链路必须有接口自动化或契约测试。
- AI 生成代码必须标注影响范围和自检说明。
- 高复杂度 Diff 需要人工 Review。

L4：

- L3 全部满足。
- 架构 Owner Review。
- 安全/SCA/SAST 无阻断项。
- 回滚方案齐全。
- 质量委员会或授权角色审批。

## 8. 测试资产中心

测试资产中心负责维护测试用例、测试数据、自动化映射和执行结果。

### 8.1 测试用例来源

- 人工创建。
- 从 PRD 自动生成。
- 从历史缺陷反推。
- 从接口契约生成。
- 从 PR Diff 影响面生成。
- 从线上事故复盘沉淀。

### 8.2 测试用例分层

- 需求验收用例
- 接口用例
- 契约用例
- E2E 用例
- 异常用例
- 兼容性用例
- 性能场景
- 混沌场景
- UAT 业务样例

### 8.3 测试用例数据结构

```json
{
  "case_id": "TC-2026-0001",
  "project_id": "P-2026-001",
  "requirement_id": "REQ-2026-0001",
  "title": "",
  "type": "api|e2e|contract|performance|chaos|uat",
  "priority": "P0|P1|P2|P3",
  "risk_level": "L1|L2|L3|L4",
  "precondition": "",
  "steps": [],
  "expected_result": "",
  "test_data_refs": [],
  "automation_binding": {
    "framework": "",
    "suite": "",
    "case_key": ""
  },
  "status": "draft|reviewed|automated|deprecated"
}
```

### 8.4 用例生成流程

1. 测试编排 Agent 读取需求规则卡、PR 风险摘要、影响面分析。
2. 生成用例候选集。
3. QA Owner 审阅。
4. 通过后进入测试资产库。
5. 自动化可绑定已有脚本或生成待自动化任务。

### 8.5 用例维护机制

- 用例必须关联需求、PR、缺陷或事故来源。
- 自动化用例必须绑定执行框架和 case key。
- 失败用例必须归因。
- 长期无命中或重复用例进入清理候选。
- 线上逃逸缺陷必须反向补充用例。

### 8.6 大型项目测试资产组织方式

测试资产需要按项目、系统、需求、风险和自动化状态多维组织：

- 项目级测试计划：覆盖整个项目的测试范围、阶段、资源和门禁。
- 系统级测试集：每个系统维护自己的接口、契约、回归和异常用例。
- 跨系统链路用例：覆盖订单、优惠、库存、支付、履约等端到端链路。
- 风险专项用例：金额一致性、库存并发、数据回补、回滚、灰度监控。
- UAT 样例：面向业务验收，强调业务输入、期望结果和签收证据。
- 自动化映射：每个可自动化用例必须绑定框架、Suite、Case Key、执行环境。

用例生成不是一次性动作，需要持续维护：

- PRD 变更时增量生成或标记受影响用例。
- PR Diff 变化时推荐补充回归用例。
- 自动化失败后标记用例稳定性。
- 线上缺陷复盘后反向生成缺失用例。
- 低价值、重复、长期不执行的用例进入清理候选。

## 9. 自动化测试集成

### 9.1 集成方式

平台不替代自动化框架，而是做编排和结果汇聚。

支持：

- REST API 触发自动化任务。
- CI Job 触发。
- Jenkins/GitLab CI/GitHub Actions 回调。
- 自研自动化平台 API。

### 9.2 自动化执行流程

1. 测试编排 Agent 生成必跑测试集。
2. 平台根据 case binding 找到自动化 suite。
3. 触发执行。
4. 自动化平台回传结果。
5. 平台聚合：
   - 通过率
   - 失败用例
   - 失败原因
   - 日志链接
   - 截图/报告链接
6. 失败归因 Agent 或 QA Owner 归因。
7. 结果进入发布门禁。

## 11. Agent 体系设计

### 11.1 Agent 类型

平台应支持三类 Agent：

1. 阶段 Agent：绑定 SDLC 阶段，例如需求澄清、测试编排。
2. 角色 Agent：绑定角色职责，例如 QA Owner Agent、发布经理 Agent。
3. 工具 Agent：绑定外部平台操作，例如 Sonar Evidence Agent、Automation Runner Agent。

### 11.4 各 Agent 背后需要的平台能力

Agent 不是一个提示词，而是依赖平台能力、数据资产、工具权限和治理机制的工程组件。每个 Agent 背后至少需要以下能力：

| Agent | 需要的平台能力 | 依赖数据 | 可调用工具 | 人工确认点 |
| --- | --- | --- | --- | --- |
| 需求澄清 Agent | 文档解析、原文引用、规则抽取、歧义识别、版本对比 | PRD、历史需求、业务规则、UAT 反馈 | 文档库、需求平台、知识库 | 业务规则、验收标准、歧义点 |
| 方案评审 Agent | 系统拓扑、依赖分析、接口契约比对、数据影响分析 | 系统清单、仓库、接口、数据血缘、事故库 | CMDB、接口平台、数据血缘、架构库 | L3/L4 方案、跨系统依赖 |
| Coding Agent | 代码上下文、代码规范、授权边界、单测生成、自检说明 | 需求规则卡、方案、代码仓库、规范 | Git、IDE 插件、单测框架 | 核心代码、权限、安全、资金相关代码 |
| PR Diff Agent | Diff 解析、影响面识别、风险评级、Sonar 汇聚、Review 清单 | PR、Diff、Sonar、覆盖率、历史缺陷 | Git、Sonar、CI、SAST、SCA | 高风险 PR、跨仓库变更 |
| 测试编排 Agent | 测试策略生成、用例生成、数据计划、自动化推荐、失败归因 | 需求规则卡、PR 风险、影响面、历史用例、缺陷 | 数据银行、自动化平台、性能平台、混沌工程 | L3/L4 测试覆盖、失败归因 |
| UAT Agent | 业务样例生成、验收包、遗留风险说明、签收流 | 验收标准、测试报告、业务规则、样例数据 | UAT 平台、数据银行、报告中心 | 业务签收、遗留风险接受 |
| 发布决策 Agent | 证据汇总、门禁评估、灰度策略、回滚校验、审批流 | PR 门禁、测试结果、UAT、监控、缺陷、回滚方案 | 发布系统、监控、审批、缺陷平台 | 例外发布、全量发布、L4 发布 |
| 复盘沉淀 Agent | 事故归因、规则沉淀、用例反哺、知识更新、Agent 评测 | 线上监控、缺陷、发布报告、Agent 输出、人工反馈 | 缺陷库、知识库、指标平台 | 规则变更、Agent 权限变更 |

### 11.5 Agent 运行与治理要求

- Agent 输入必须来自平台已授权数据，不允许自由读取未授权系统。
- Agent 输出必须结构化，包含结论、证据引用、置信度、风险、建议动作。
- Agent 不能直接修改生产策略或放行高风险发布。
- Agent 配置变更必须经过评测集验证。
- Agent 工具权限必须可配置、可审批、可审计。
- Agent 运行结果必须允许人工采纳、驳回、补充和反馈。
- Agent 的误判、漏判、误阻断需要进入质量运营复盘。

### 11.6 Agent 管理页面产品能力

Agent 管理不能只是展示列表，需要支持：

- Agent 定义维护：名称、阶段、角色、Owner、版本、状态。
- Prompt 维护：系统提示词、业务约束、禁止事项。
- Schema 维护：输入 Schema、输出 Schema。
- 工具权限维护：工具、权限级别、是否需要审批、作用范围。
- 策略规则维护：阻断规则、升级规则、例外规则。
- 评测用例维护：正常、异常、缺证据、误放行、误阻断场景。
- 试运行：选择项目、需求、PR 或测试计划进行模拟。
- 发布：配置评测通过后发布新版本。
- 回滚：回退到上一稳定版本。
- 审计：记录谁改了什么、为什么改、何时生效。

### 11.2 Agent 配置项

每个 Agent 必须可维护：

- 基础信息
- 系统提示词
- 输入 Schema
- 输出 Schema
- 工具权限
- 策略规则
- 评测用例
- 角色权限
- 版本记录
- 灰度状态
- 回滚版本

### 11.3 Agent 运行原则

- Agent 输出不能直接等同于最终审批。
- 高风险结论必须引用证据。
- 工具执行权限必须分级。
- 生产发布类动作必须人工审批。
- Agent 配置变更必须跑评测集。
