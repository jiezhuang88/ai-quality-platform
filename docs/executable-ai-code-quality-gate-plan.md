# AI 大量提交代码场景下的质量门禁可执行方案

## 1. 目标

在 AI coding 大量提升代码产出速度的情况下，测试工程师需要建立一套可执行、可自动化、可审计的质量门禁体系，确保 AI 生成代码不能未经验证直接进入主干或上线环境。

核心目标：

- 将 AI 代码变更纳入统一准入流程。
- 根据风险等级动态调整测试和门禁强度。
- 用自动化测试、静态扫描、安全扫描和 AI 辅助评审提高反馈速度。
- 让每次代码合并都有明确质量证据。
- 将逃逸缺陷持续转化为测试用例、Prompt 约束和门禁规则。

## 2. 总体阶段流程

```text
需求/变更提出
    |
    v
AI 辅助编码
    |
    v
提交 PR
    |
    v
变更准入检查
    |
    v
风险等级评估
    |
    v
按风险执行自动化测试和扫描
    |
    v
AI 辅助测试分析和代码评审
    |
    v
人工 Review
    |
    v
质量门禁判定
    |
    v
合并 / 阻断 / 返工
    |
    v
发布后监控和缺陷复盘
```

## 3. 阶段执行方案

### 阶段一：变更准入

每个 AI 参与的代码提交都必须先进入 PR 流程，不允许直接合并主分支。

PR 必须包含：

- 本次变更目标。
- AI 参与范围。
- 改动文件和影响模块。
- 是否涉及核心业务链路。
- 是否涉及权限、支付、订单、数据、隐私、安全。
- 是否新增依赖。
- 测试说明。
- 回滚方案。
- 已知剩余风险。

门禁规则：

- 缺少变更说明，阻断。
- AI 参与但未说明 AI 生成范围，阻断。
- 涉及核心链路但无测试计划，阻断。
- 涉及数据库或配置变更但无回滚说明，阻断。

### 阶段二：风险等级评估

测试工程师或系统根据变更内容自动/半自动评估风险等级。

评估维度：

| 维度 | 低风险 | 中风险 | 高风险 |
| --- | --- | --- | --- |
| 用户影响 | 内部工具或非核心页面 | 普通用户功能 | 核心用户链路 |
| 数据影响 | 不改数据 | 读写普通业务数据 | 订单、支付、权限、隐私数据 |
| 技术复杂度 | 单模块、小改动 | 多模块协作 | 跨系统、异步、并发、状态机 |
| 可回滚性 | 直接回滚 | 需要配置或人工处理 | 数据不可逆或恢复复杂 |
| 安全影响 | 无安全边界 | 普通鉴权/校验 | 权限、认证、敏感数据、外部输入 |
| AI 自主度 | AI 辅助小修改 | AI 生成部分逻辑 | AI 生成核心逻辑或大量代码 |

风险等级：

- 低风险：文案、样式、小范围重构、非核心工具。
- 中风险：接口逻辑、数据转换、表单校验、普通业务规则。
- 高风险：权限、支付、订单、登录、数据迁移、隐私、安全、核心链路。
- 严重风险：不可逆数据变更、合规相关、生产安全、金融金额、批量任务。

### 阶段三：按风险决定门禁强度

#### 低风险门禁

适用场景：

- 文案修改。
- UI 样式微调。
- 非核心代码重构。
- 内部工具小改动。

必须通过：

- PR 信息完整。
- 单元测试或相关快照测试通过。
- 静态代码扫描无高危问题。
- 密钥扫描通过。
- 人工 Review 通过。

可选：

- AI 辅助检查边界影响。
- 小范围回归测试。

阻断条件：

- 无 PR 说明。
- 静态扫描高危问题。
- 出现敏感信息。
- Review 发现业务语义不清。

#### 中风险门禁

适用场景：

- API 行为变化。
- 表单校验。
- 数据转换。
- 普通业务规则。
- 多模块调用。

必须通过：

- 低风险全部门禁。
- 单元测试覆盖正常、异常、边界场景。
- 接口测试通过。
- 集成测试或契约测试通过。
- 覆盖率不下降，或下降有明确说明。
- AI 辅助生成测试建议并由人工确认。

阻断条件：

- 有行为变更但无对应测试。
- API 入参/出参变化但无契约测试。
- 异常路径未覆盖。
- 覆盖率下降且无解释。

#### 高风险门禁

适用场景：

- 权限。
- 支付。
- 订单。
- 登录认证。
- 数据库迁移。
- 隐私数据。
- 核心用户旅程。
- 安全边界。

必须通过：

- 中风险全部门禁。
- 关键链路 E2E 测试通过。
- 安全测试通过。
- 权限负向测试通过。
- 依赖扫描通过。
- 回滚方案明确。
- 发布监控指标明确。
- 测试工程师专项 Review。
- 开发负责人或架构负责人 Review。

阻断条件：

- 涉及权限但没有越权/负向测试。
- 涉及支付/订单但没有 E2E。
- 涉及数据库但没有回滚或迁移验证。
- 新增依赖但未做依赖漏洞扫描。
- 安全扫描存在高危问题。
- 无发布后监控方案。

#### 严重风险门禁

适用场景：

- 不可逆数据变更。
- 生产批量任务。
- 金融金额计算。
- 合规/隐私强相关。
- 大规模架构调整。

必须通过：

- 高风险全部门禁。
- 独立测试计划评审。
- 灰度或分阶段发布方案。
- 数据备份和恢复演练。
- 生产监控和告警配置。
- 明确审批人。
- 发布后复盘机制。

阻断条件：

- 无灰度方案。
- 无备份恢复方案。
- 无业务负责人确认。
- 无可观测性验证。

## 4. 自动化测试和扫描建设

### CI/CD 自动门禁

每次 PR 自动执行：

```text
代码提交
  -> 单元测试
  -> 接口/集成测试
  -> 静态代码扫描
  -> 密钥扫描
  -> 依赖漏洞扫描
  -> 覆盖率检查
  -> 风险规则检查
  -> 生成质量报告
```

推荐自动化检查项：

| 检查项 | 目的 | 阻断条件 |
| --- | --- | --- |
| 单元测试 | 验证基础逻辑 | 测试失败 |
| 接口测试 | 验证 API 行为 | 关键接口失败 |
| 契约测试 | 防止上下游破坏 | 契约不兼容 |
| E2E 测试 | 保护核心链路 | 高风险链路失败 |
| 静态扫描 | 发现坏味道和危险代码 | 高危问题 |
| 密钥扫描 | 防止泄露 token/密码 | 任意疑似密钥 |
| 依赖扫描 | 防止引入漏洞依赖 | 高危漏洞 |
| 覆盖率检查 | 防止测试倒退 | 核心覆盖率下降 |
| 变更影响分析 | 自动选择回归范围 | 无法识别影响需人工确认 |

## 5. AI 辅助测试方案

AI 不直接决定是否放行，但可以辅助测试工程师提高效率。

### AI 可以做的事情

- 根据 PR diff 生成测试点。
- 根据代码变更识别影响范围。
- 根据需求生成边界条件。
- 识别缺失的异常路径。
- 生成接口测试数据。
- 分析失败日志。
- 总结 PR 风险。
- 生成回归测试建议。
- 对测试用例做质量审查。

### AI 测试分析 Prompt 模板

```text
请作为资深测试工程师分析这次代码变更。

输入：
- 需求说明
- 代码 diff
- 已有测试用例
- PR 描述

请输出：
1. 本次变更影响范围
2. 风险等级建议
3. 必须补充的测试用例
4. 必须执行的回归范围
5. 是否存在权限、安全、数据一致性、并发、异常处理风险
6. 当前测试是否足够
7. 是否建议阻断合并
```

### AI 生成测试后的人工校验

测试工程师必须确认：

- 测试是否验证业务行为，而不是验证实现细节。
- 测试是否真的能在代码出错时失败。
- 是否覆盖异常、边界、负向场景。
- Mock 是否真实。
- 是否遗漏权限、并发、数据一致性、安全场景。

## 6. 质量门禁判定规则

可以采用以下合并规则：

允许合并：

- 风险等级已确认。
- 对应风险等级门禁全部通过。
- 自动化测试通过。
- 扫描无高危问题。
- 人工 Review 通过。
- 高风险变更有回滚和监控方案。

阻断合并：

- 测试失败。
- 无测试或测试明显不足。
- 安全扫描高危。
- 密钥泄露。
- 新增依赖未扫描。
- 权限变更无负向测试。
- 核心链路变更无 E2E。
- 数据库变更无回滚。
- AI 生成代码无人 Review。
- PR 描述不完整。

需要人工裁决：

- 覆盖率下降但业务风险较低。
- AI 判断与测试工程师判断不一致。
- 变更影响范围无法自动识别。
- 存在已知风险但业务要求紧急发布。

## 7. 测试工程师职责

测试工程师不是单纯执行测试，而是质量门禁负责人。

主要职责：

- 设计风险分级规则。
- 维护质量门禁标准。
- 建设自动化测试体系。
- 审核 AI 生成测试用例质量。
- 建设核心链路回归集。
- 维护缺陷模式库。
- 推动逃逸缺陷转化为自动化用例。
- 持续优化 Prompt、Checklist 和 CI 门禁。

## 8. 落地路线图

### 第 1 阶段：建立规则

周期：1-2 周。

- 建立 PR 模板。
- 建立风险分级表。
- 明确低/中/高风险门禁。
- 要求 AI 代码必须说明 AI 参与范围。

### 第 2 阶段：接入自动化

周期：2-4 周。

- 接入单元测试。
- 接入接口测试。
- 接入静态扫描。
- 接入密钥扫描。
- 接入覆盖率检查。
- 在 CI 中自动阻断高危问题。

### 第 3 阶段：接入 AI 测试分析

周期：4-6 周。

- 用 AI 分析 PR diff。
- 用 AI 生成测试建议。
- 用 AI 总结风险。
- 测试工程师审核 AI 输出。
- 形成测试建议报告。

### 第 4 阶段：风险驱动回归

周期：6-10 周。

- 建立核心链路 E2E。
- 建立变更影响分析。
- 按改动模块自动选择回归集。
- 对高风险变更强制执行专项测试。

### 第 5 阶段：持续度量和改进

长期执行。

- 跟踪缺陷逃逸率。
- 跟踪 AI 变更失败率。
- 跟踪自动化通过率。
- 跟踪覆盖率变化。
- 每个逃逸缺陷必须补充回归用例。
- 每月复盘门禁规则是否有效。

## 9. 可执行 Checklist

每个 AI 代码 PR 合并前确认：

- [ ] PR 描述完整。
- [ ] AI 参与范围已说明。
- [ ] 风险等级已评估。
- [ ] 测试计划已补充。
- [ ] 单元测试通过。
- [ ] 接口/集成测试按风险执行。
- [ ] 核心链路 E2E 按风险执行。
- [ ] 静态扫描通过。
- [ ] 密钥扫描通过。
- [ ] 依赖扫描通过。
- [ ] 权限/安全负向测试已覆盖。
- [ ] 覆盖率无异常下降。
- [ ] AI 辅助评审已完成。
- [ ] 人工 Review 已完成。
- [ ] 高风险变更已有回滚方案。
- [ ] 高风险变更已有发布监控方案。

## 10. 最终原则

AI 代码质量门禁不是为了降低 AI 使用，而是为了让 AI 编码可以安全规模化。

测试工程师要做到：

- 用风险分级决定测试深度。
- 用自动化门禁替代人工重复检查。
- 用 AI 辅助发现测试盲区。
- 用人工判断兜住业务语义和风险决策。
- 用度量和复盘持续改进质量体系。

一句话：

**AI 负责提高代码产出速度，测试工程师负责建立一套让 AI 代码不能带病合并、不能带病上线的质量门禁系统。**
