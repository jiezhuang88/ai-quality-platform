# 01 项目空间需求文档

版本：v0.1  
日期：2026-06-15  
对应规划：大型项目可控、项目空间模型、年度项目组合管理

## 1. 背景和目标

AI Coding 提升了代码产出速度，但大型项目往往不是单仓库、单系统、单团队完成。一个项目可能同时涉及订单、优惠、库存、支付、履约、数据、算法等多个系统，也可能对应多个 Git 仓库、多个 PR、多个测试计划和多个发布批次。

项目空间的目标是提供统一项目控制面，让研发组织可以围绕一个业务项目管理需求、系统、仓库、PR、测试、UAT、发布和复盘证据。

## 2. 用户角色

| 角色 | 核心诉求 |
| --- | --- |
| 项目负责人 | 看清项目范围、风险、阻断项和发布状态 |
| 产品经理 | 导入 PRD，确认需求规则和验收标准 |
| 架构师/技术负责人 | 维护系统影响面、架构风险、发布依赖 |
| 研发工程师 | 关联仓库、PR、AI 自检说明和代码证据 |
| 测试负责人 | 维护测试计划、测试资产、执行证据和风险结论 |
| UAT/业务验收 | 查看验收包、样例数据、遗留风险 |
| 质量运营 | 统计项目组合、风险分布、证据完整率 |

## 3. 业务流程

1. 创建项目空间。
2. 选择业务域、项目类型、预计上线窗口、风险初判。
3. 关联多个系统和系统 Owner。
4. 为每个系统关联一个或多个仓库。
5. 导入或登记 PRD、需求链接、业务验收标准。
6. 建立项目质量计划，确定 L1-L4 风险等级。
7. 后续 PR、测试、UAT、发布证据全部挂接到项目空间。
8. 项目发布后进入复盘和资产沉淀。

## 4. 功能需求

### 4.1 项目创建

系统应支持创建项目，并维护以下字段：

| 字段 | 说明 |
| --- | --- |
| 项目名称 | 业务可识别名称 |
| 项目编号 | 平台生成，支持外部项目号映射 |
| 业务域 | 订单、优惠、库存、支付、履约、数据、算法等 |
| 项目类型 | 新功能、改造、重构、数据修复、大促、平台建设 |
| 预计上线窗口 | 支持日期和发布批次 |
| 初始风险等级 | L1-L4，可后续调整 |
| 项目负责人 | 必填 |
| 质量负责人 | L3/L4 必填 |
| 状态 | 草稿、进行中、待发布、已发布、已归档 |

### 4.2 多系统管理

一个项目必须支持关联多个系统。

系统信息包括：

- 系统名称。
- 系统编码。
- 所属业务域。
- 系统 Owner。
- 是否核心链路系统。
- 上下游依赖。
- 发布窗口约束。

### 4.3 多仓库管理

每个系统可关联多个仓库。

仓库信息包括：

- Git 地址。
- 默认分支。
- 代码语言。
- 关联 CI 流水线。
- Sonar project key。
- 自动化测试 suite 映射。
- 仓库 Owner。

### 4.4 项目总览

项目空间首页应展示：

- 项目阶段进度。
- 当前风险等级。
- 关联系统数量。
- 关联仓库数量。
- PR 数量和门禁状态。
- 测试用例候选、已入库、已执行数量。
- 证据完整率。
- 阻断项列表。
- 下一步待办。

## 5. 数据对象

核心对象：

- Project。
- ProjectSystem。
- Repository。
- Stakeholder。
- ProjectStage。
- ProjectRiskProfile。
- ProjectBlocker。

## 6. 规则要求

- L3/L4 项目必须至少关联一个质量负责人。
- 项目进入 PR 阶段前，必须至少完成系统和仓库登记。
- 多仓库项目必须能识别跨仓库 PR 集合。
- 项目归档前必须完成发布结论或取消原因登记。

## 7. 一期范围

一期必须支持：

- 创建项目。
- 维护多系统、多仓库。
- 查看项目总览。
- 挂接 PRD、PR、测试和证据对象。
- 支持 L1-L4 风险等级字段。

一期暂不强制实现：

- 自动从企业 CMDB 同步系统。
- 自动从 Git 平台发现仓库。
- 复杂项目组合资源排期。

## 8. 验收标准

- 一个项目可关联不少于 5 个系统、20 个仓库。
- 项目总览能展示 PRD、PR、测试、证据状态。
- L3/L4 项目缺少质量负责人时系统提示阻断。
- 多仓库项目能按仓库维度查看 PR 和质量证据。
