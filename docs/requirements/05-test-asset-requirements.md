# 05 测试资产生成与维护需求文档

版本：v0.1  
日期：2026-06-15  
对应规划：专业测试用例生成、Review、入库、自动化绑定、缺陷反哺

## 1. 背景和目标

AI 可以快速生成大量测试用例，但没有 Review、去重、入库和执行证据关联的用例只是一次性文本。测试资产模块的目标是把 AI 生成候选变成可维护、可复用、可度量的正式测试资产。

## 2. 业务流程

1. 从已确认需求规则卡、PR 风险摘要或缺陷复盘触发用例生成。
2. 测试编排 Agent 生成候选用例。
3. QA Review 候选用例。
4. 对用例执行批准、退回、修改、合并、废弃。
5. 批准后的用例进入测试资产库。
6. 对 P0/P1 用例绑定自动化。
7. 执行结果和失败归因进入证据中心。
8. 线上逃逸缺陷反向生成用例更新候选。

## 3. 用例生成 Skill 要求

测试用例生成必须使用专业 Skill，覆盖：

- 等价类。
- 边界值。
- 决策表。
- 状态迁移。
- 异常和降级。
- 幂等和重试。
- 数据一致性。
- 契约测试。
- 回滚验证。
- 性能和容量。
- 混沌场景。

## 4. 用例字段

候选用例和正式用例均应包含：

- 用例 ID。
- 标题。
- 测试目标。
- 用例类型。
- 优先级。
- 风险等级。
- 关联需求规则卡。
- 关联 PR。
- 影响系统。
- 前置条件。
- 测试数据。
- 操作步骤。
- 预期结果。
- 明确断言。
- 负向或边界说明。
- 自动化建议。
- Review 状态。
- 入库版本。
- 执行证据。

为了支撑一个大型项目中上百到上千条测试用例，用例还应具备以下管理字段：

- 所属测试集或测试包。
- 所属业务域。
- 所属系统。
- 所属需求规则卡。
- 所属 PR 或变更批次。
- Review 批次。
- Review 负责人。
- 自动化绑定状态。
- 最近执行状态。
- 最近失败原因。
- 标签，如核心链路、金额、库存、回滚、性能、混沌、UAT。

## 5. Review 流程

候选用例状态：

- candidate：候选。
- in_review：Review 中。
- returned：退回修改。
- approved：批准。
- rejected：废弃。
- baselined：已入库。

Review 结论必须记录：

- Review 人。
- Review 意见。
- 处理动作。
- 处理时间。
- 是否需要自动化。
- 是否核心链路 P0/P1。

### 5.1 大规模用例管理视图

测试资产页面不得采用单一平铺列表承载全部用例。系统应提供结构化管理视图：

| 视图 | 目的 |
| --- | --- |
| 资产概览 | 展示总数、候选、待 Review、已入库、已自动化、失败用例数量 |
| Review 队列 | 按状态、优先级、类型、系统聚合待处理候选 |
| 用例分组 | 按系统、需求规则卡、用例类型、优先级、自动化状态分组 |
| 用例详情 | 展示步骤、测试数据、断言、追溯、Review 记录和自动化绑定 |
| 执行视图 | 按 suite、最近执行状态、失败原因查看执行证据 |

### 5.2 筛选和批量操作

系统应支持：

- 按状态筛选：candidate、returned、reviewed、baselined、rejected。
- 按优先级筛选：P0、P1、P2。
- 按用例类型筛选：api、e2e、contract、performance、chaos、uat。
- 按影响系统筛选。
- 按自动化状态筛选：未绑定、已绑定、执行失败。
- 按关键词搜索标题、目标、断言和测试数据。
- 对同一 Review 批次进行批量通过、批量退回、批量入库。

批量操作必须保留审计记录，并允许 QA 对单条用例进入详情后精细 Review。

## 6. 自动化绑定

入库用例可绑定：

- API 自动化 suite。
- E2E 自动化 suite。
- 契约测试。
- 性能测试脚本。
- 混沌实验。
- 数据银行数据模板。

P0/P1 核心链路用例应优先绑定自动化。

## 7. 缺陷反哺

线上逃逸缺陷复盘后，系统应支持：

- 关联缺陷。
- 识别缺失用例。
- 生成补充用例候选。
- 更新已有用例。
- 形成 Agent Eval 样例。

## 8. 一期范围

一期必须支持：

- 从需求规则卡生成候选用例。
- QA Review。
- 批准入库。
- 自动化绑定登记。
- 执行结果登记。
- 用例按状态、优先级、类型、系统进行结构化展示。
- 用例详情展开，展示步骤、测试数据、断言、追溯和 Review 记录。

一期暂不强制实现：

- 自动去重全量测试资产。
- 自动触发自动化执行。
- 从真实缺陷系统自动同步缺陷。
- 真正跨项目的企业级用例检索和推荐。

## 9. 验收标准

- 候选用例必须经过 Review 才能入库。
- 入库用例必须能追溯需求规则卡或 PR。
- P0/P1 用例必须提示自动化绑定要求。
- 执行失败必须能记录失败原因和归属。
- 页面不得只平铺展示所有用例，必须支持分组、筛选和详情阅读。
- 当项目存在上百条用例时，Review 人可以先按系统、优先级、类型缩小范围，再处理单条或批量用例。
