# 02 需求导入与规则卡需求文档

版本：v0.1  
日期：2026-06-15  
对应规划：PRD 导入、需求规则卡、需求澄清 Agent

## 1. 背景和目标

AI Coding 下代码生成速度变快，但如果需求不清晰，风险会更早、更快地放大。需求导入模块的目标不是简单存文件，而是把 PRD、业务规则、验收标准、歧义点和影响面转成可评审、可追踪、可生成测试资产的结构化输入。

## 2. 输入来源

系统应支持以下导入方式：

| 来源 | 一期能力 | 后续增强 |
| --- | --- | --- |
| 文件上传 | Markdown、Word、PDF、TXT | 自动解析表格、图片、流程图 |
| 文本粘贴 | 支持直接粘贴 PRD 内容 | 自动识别章节和规则 |
| 飞书链接 | 登记链接和标题 | 接入飞书 API 读取正文 |
| Confluence/Jira 链接 | 登记链接和元数据 | 接入 API 同步状态 |
| URL | 登记外部需求地址 | 按权限抓取内容 |

PRD 导入入口必须采用级联方式，避免把文件、飞书、Jira、URL 等不同来源混在一个单层下拉中。

级联层级：

| 第一级：来源大类 | 第二级：具体来源 | 第三级：导入动作 |
| --- | --- | --- |
| 本地上传 | Markdown、Word、PDF、TXT | 选择文件、读取预览、上传快照 |
| 手动录入 | 粘贴文本、会议纪要摘录 | 粘贴内容、填写来源名称、解析 |
| 协作文档 | 飞书文档、Confluence 页面 | 粘贴链接、登记 token、后续 API 拉取 |
| 研发协同 | Jira Story、需求单、缺陷单 | 粘贴链接、登记外部 ID、同步状态 |
| 外部链接 | 普通 URL、知识库页面 | 登记地址、保存元数据、后续抓取 |

级联选择要求：

- 第一级决定页面展示哪些字段。
- 第二级决定 `external_provider`、解析策略和权限提示。
- 第三级决定是上传文件、读取本地文本、登记链接还是调用外部 Adapter。
- 切换来源类型时，系统不得丢失已输入内容，应保留草稿。
- 对接能力未完成时，页面必须明确标识“当前为登记链接，后续通过 Adapter 同步正文”。

## 3. 业务流程

1. 用户在项目空间中导入 PRD。
2. 平台保存原始来源和版本。
3. 需求澄清 Agent 解析需求，生成规则卡候选。
4. 产品、研发、测试共同 Review。
5. 对歧义点进行确认、驳回或补充。
6. 规则卡确认后进入后续方案评审、PR 门禁和测试生成。

## 4. 功能需求

### 4.1 PRD 来源管理

每个 PRD 来源必须记录：

- 来源类型。
- 来源大类。
- 具体来源。
- 导入动作。
- 来源地址或文件名。
- 上传人。
- 上传时间。
- 版本号。
- 原文快照。
- 解析状态。

### 4.2 需求规则卡

规则卡字段包括：

| 字段 | 说明 |
| --- | --- |
| 规则 ID | 平台生成 |
| 规则标题 | 简明描述业务规则 |
| 原文引用 | 指向 PRD 原文段落 |
| 业务域 | 订单、优惠、库存、支付等 |
| 影响系统 | 可多选 |
| 用户场景 | 正常、异常、边界、回滚 |
| 输入条件 | 触发条件和前置约束 |
| 处理逻辑 | 核心业务逻辑 |
| 输出结果 | 用户可见结果、数据状态、事件消息 |
| 验收标准 | 可验证的业务期望 |
| 风险等级建议 | L1-L4 |
| 状态 | 候选、待确认、已确认、已废弃 |

### 4.3 歧义点管理

Agent 发现以下情况时必须生成歧义点：

- 条件不完整。
- 例外场景缺失。
- 数据口径不清。
- 与历史规则冲突。
- 缺少验收标准。
- 涉及多个系统但未明确责任边界。

歧义点必须支持负责人、处理状态、处理结论和确认记录。

### 4.4 版本对比

同一项目允许导入多个 PRD 版本，系统应支持：

- 查看版本列表。
- 标记当前生效版本。
- 查看新增、修改、删除的规则卡。
- 识别需求变更对测试资产和 PR 风险的影响。

## 5. Agent 能力要求

需求澄清 Agent 必须输出：

- 需求摘要。
- 业务规则卡候选。
- 歧义点。
- 影响系统建议。
- 风险等级建议。
- 测试关注点建议。
- 原文证据引用。

Agent 输出不得直接成为正式规则，必须由产品、研发或测试确认。

## 6. 一期范围

一期必须支持：

- 级联式 PRD 导入入口。
- 文件、文本、飞书链接、Confluence/Jira/URL 链接登记。
- 规则卡候选生成。
- 歧义点 Review。
- 规则卡确认。
- 规则卡与测试用例、PR、证据关联。

一期暂不强制实现：

- 自动读取飞书正文。
- 复杂图片和流程图解析。
- 跨版本自动影响面全量计算。

## 7. 验收标准

- PRD 导入后能生成不少于一组规则卡候选。
- PRD 导入方式必须通过“来源大类 -> 具体来源 -> 导入动作”三级选择完成。
- 不同来源类型只能展示相关字段，例如文件上传展示文件选择，飞书/Jira 展示链接和外部 ID。
- 每条规则卡必须能追溯原文来源。
- 歧义点必须支持确认、补充、关闭。
- 已确认规则卡可被测试用例生成模块引用。
