# AI 时代研发协同与质量工程平台 PRD

版本：v1.0  
关联主文档：[AI 时代研发协同与质量工程平台整体设计文档](ai-dev-collaboration-platform-design.md)

## 1. 产品定位

### 1.1 产品一句话

面向产品、研发、测试、发布、业务验收的研发协同平台，通过流程编排、工具集成、AI Agent 辅助和质量风险门禁，让 AI Coding 时代的软件变更可评估、可验证、可追踪、可复盘。

### 1.2 目标用户

| 角色 | 核心诉求 |
| --- | --- |
| CTO/研发负责人 | 看清 AI 对交付效率、质量、稳定性的真实影响 |
| 项目/产品负责人 | 确保需求规则和验收标准被研发、测试、UAT 一致理解 |
| 研发负责人 | 降低 AI 生成代码带来的 Review 压力和隐性风险 |
| 开发工程师 | 在 PR 阶段明确风险、测试要求和阻断原因 |
| 测试工程师 | 从执行测试升级为质量策略、门禁和证据编排 |
| 发布/运维负责人 | 发布前看到灰度、监控、回滚和风险证据是否完整 |
| 业务验收团队 | 聚焦业务规则是否被正确实现，而不是追踪技术细节 |

### 1.3 产品目标

1. 建立统一研发作业流，覆盖需求、设计、AI 编码、PR、测试验证、UAT、发布、复盘。
2. 建立风险驱动的门禁体系，低风险快速通过，高风险自动加严。
3. 集成现有工具平台，复用数据银行、自动化框架、性能平台、混沌工程、Sonar、覆盖率、缺陷和监控数据。
4. 引入 AI Agent 辅助需求澄清、方案评审、PR 风险识别、测试推荐、失败归因和缺陷复盘。
5. 建立证据中心和研发经营看板，让每次变更都有可审计证据。

## 2. 范围定义

### 2.1 MVP 范围

| 模块 | MVP 能力 |
| --- | --- |
| 变更工作台 | 展示变更、风险、状态、责任人、证据完整度 |
| 风险分级 | 根据业务链路、数据风险、影响范围、AI 参与度、测试缺口计算风险等级 |
| PR 门禁 | 接入 PR、Sonar、单测覆盖率、接口自动化、人工 Review 结论 |
| 证据中心 | 聚合需求、代码、测试、UAT、发布证据索引 |
| 报告导出 | 导出变更质量报告和发布准入报告 |
| 研发 cockpit | 展示 Lead Time、AI CFR、缺陷逃逸率、高风险阻断率、核心链路 P95/P99 |
| 试点链路 | 订单交易核心链路：全场券、商品券、库存、支付前金额一致性 |

### 2.2 非 MVP 范围

- 不替代 Jira、Git、CI/CD、监控、缺陷系统。
- 不在第一阶段做全自动发布决策。
- 不在第一阶段让 AI 自动修改门禁规则。
- 不在第一阶段覆盖所有业务域，先聚焦订单交易核心链路。

## 3. 用户旅程

### 3.1 高风险 AI 代码变更旅程

1. 产品提交需求和验收标准。
2. 平台调用需求澄清 Agent 抽取业务规则和风险点。
3. 研发提交技术方案和 AI Coding 使用说明。
4. 平台在 PR 创建后触发 PR Diff Agent。
5. 风险引擎识别订单、优惠、库存、支付前金额校验，判定 L3 高风险。
6. 平台自动要求接口自动化、E2E、数据银行、性能基线、UAT、灰度监控证据。
7. QA 在平台内查看推荐测试策略，补充探索测试。
8. 业务在 UAT 阶段确认全场券、商品券规则。
9. 发布负责人查看证据完整度、灰度和回滚方案。
10. 发布后监控数据和缺陷复盘反哺知识库。

## 4. 功能需求

### 4.1 变更工作台

**需求描述**

提供单个变更的统一视图，汇总需求、设计、PR、风险、门禁、测试、UAT、发布和复盘信息。

**功能点**

- 创建或同步变更。
- 关联需求 ID、PR ID、发布单、缺陷。
- 展示当前阶段、负责人、阻断项。
- 展示风险等级和风险因子。
- 展示证据完整度。
- 支持导出报告。

**验收标准**

- 用户可在一个页面看到变更从需求到发布的状态。
- 高风险变更缺少关键证据时，页面必须明确显示阻断原因。
- 变更状态变化必须记录时间、操作人和来源系统。

### 4.2 风险分级

**需求描述**

平台根据业务链路、代码影响面、数据一致性、AI 不确定性、历史缺陷、测试缺口、发布保护等维度计算风险等级。

**功能点**

- 自动读取 PR diff、改动文件、服务、接口、历史缺陷。
- 支持人工修正风险等级。
- 风险等级调整必须填写原因。
- 风险等级映射门禁策略。

**验收标准**

- 涉及订单、优惠、库存、支付的变更默认不得低于 L3。
- 涉及不可逆数据变更、批量金额修正、核心交易一致性的变更默认 L4。
- 人工降级必须留审计记录。

### 4.3 门禁策略

**需求描述**

不同风险等级匹配不同门禁要求。

| 风险等级 | 必须门禁 |
| --- | --- |
| L1 | PR Review、静态扫描、单测 |
| L2 | L1 + 接口自动化、依赖扫描、AI Review |
| L3 | L2 + E2E、数据银行、性能基线、UAT、灰度监控 |
| L4 | L3 + 回滚演练、例外审批、发布负责人确认 |

**验收标准**

- 缺少必须门禁证据时，平台状态为 Blocked。
- 门禁通过后，平台生成可导出的发布准入报告。
- 例外审批必须记录风险、审批人、有效期和复盘任务。

### 4.4 AI Agent 协同

**需求描述**

通过多个 Agent 辅助研发协同，不替代人工最终决策。

**功能点**

- 需求澄清 Agent：抽取业务规则、验收点、歧义。
- 方案评审 Agent：识别架构、依赖、数据一致性风险。
- PR Diff Agent：识别影响范围和风险等级建议。
- 测试策略 Agent：推荐测试范围、数据需求、回归集。
- 失败归因 Agent：分析 CI、自动化、日志失败原因。
- 复盘 Agent：把缺陷转化为用例、规则、知识库条目。

**验收标准**

- Agent 输出必须展示依据、置信度和建议动作。
- 高风险变更不能由 Agent 自动放行。
- 人工采纳或驳回 Agent 建议必须记录。

### 4.5 工具集成

**需求描述**

平台统一编排已有工具能力。

**优先接入**

1. Git/PR。
2. Sonar。
3. 单测覆盖率。
4. 接口自动化。
5. 缺陷系统。
6. 数据银行。
7. 性能平台。
8. 混沌工程。
9. 监控告警。
10. CI/CD。

**验收标准**

- 接入工具必须返回任务状态、报告链接、关键指标。
- 原始报告可保留在源系统，平台保存索引和摘要。
- 工具失败时不能默认通过，需要展示异常原因。

### 4.6 研发 cockpit

**需求描述**

为管理层提供研发效率、质量、稳定性、AI 代码风险的经营视图。

**指标**

- Lead Time。
- Deployment Frequency。
- Change Failure Rate。
- MTTR。
- AI Change Failure Rate。
- Defect Escape Rate。
- High Risk Gate Hit Rate。
- Evidence Completeness。
- 核心链路 P95/P99。
- 遗留缺陷数量。

**验收标准**

- 指标可按团队、业务域、服务、风险等级、AI 参与程度筛选。
- 可区分 AI 参与变更与非 AI 参与变更。
- 可从指标下钻到具体变更和证据。

## 5. 非功能需求

| 分类 | 要求 |
| --- | --- |
| 可用性 | 核心工作台可用性不低于 99.5% |
| 性能 | 单个变更详情页 2 秒内打开，报告导出 10 秒内完成 |
| 安全 | 支持 SSO、RBAC、审计日志、敏感字段脱敏 |
| 可扩展 | 工具接入采用适配器模式，新增工具不影响核心流程 |
| 可审计 | 所有门禁、审批、Agent 输出、人工调整必须留痕 |
| 可配置 | 风险规则、门禁策略、流程阶段可配置 |
| 数据保留 | 变更证据、审批和报告至少保留 2 年，具体按公司合规要求调整 |

## 6. 关键页面

1. 平台总览。

3. 风险分级页。

5. Agent 分析页。
6. 工具任务页。
7. UAT 验收页。
8. 发布准入页。
9. 缺陷复盘页。

## 7. 成功指标

| 阶段 | 指标 |
| --- | --- |
| MVP | 订单交易链路高风险变更 80% 以上进入平台治理 |
| MVP | L3/L4 变更证据完整率达到 90% |
| 阶段 2 | 高风险 PR 自动识别准确率达到 80% 以上 |
| 阶段 2 | 测试策略推荐被采纳率达到 50% 以上 |
| 阶段 3 | 缺陷逃逸率较试点前下降 |
| 阶段 3 | AI 参与变更失败率可被稳定追踪并下降 |
